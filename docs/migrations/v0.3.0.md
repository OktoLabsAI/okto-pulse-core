# KG Schema Migration v0.2.0 → v0.3.0

**Spec:** R1 — Modelo de dados: relevance_score + remoção de validation_status
(`a3023ed2-7458-4e86-8494-fac458eea597` no board Okto Pulse Evolution)

**Status:** R1 implementado em 2026-04-18.

## O que muda

| Coluna do nó                | v0.2.0 | v0.3.0  | Notas                                   |
|-----------------------------|--------|---------|-----------------------------------------|
| `validation_status STRING`  | sim    | **não** | Removida — substituída por `relevance_score` |
| `corroboration_count INT64` | sim    | **não** | Removida — dead code, nunca incrementada |
| `relevance_score DOUBLE`    | não    | **sim** | Score contínuo, default 0.5             |
| `query_hits INT64`          | não    | **sim** | Contador de hits, default 0             |
| `last_queried_at STRING`    | não    | **sim** | ISO 8601 do último hit, default null    |

`source_confidence`, `embedding`, `created_at`, `superseded_*` permanecem.

## Estratégia de migração

Kùzu v0.6 não suporta `ALTER TABLE DROP COLUMN`, então a migração combina
duas estratégias por tabela de nó:

1. **`recreate`** — quando a tabela ainda tem `validation_status` /
   `corroboration_count`: dump em RAM via `MATCH (n:Type) RETURN ...` →
   `DROP TABLE Type` → `CREATE NODE TABLE Type (...)` com o DDL novo →
   `CREATE` linha-a-linha mapeando os defaults v0.3.0 (`relevance_score=0.5`,
   `query_hits=0`, `last_queried_at=NULL`). Embeddings são preservados.
2. **`alter`** — quando a tabela já está em v0.3.0 (boards bootstrapped em
   ambiente fresh) ou só falta as colunas novas: `ALTER TABLE ... ADD ...` +
   backfill `WHERE col IS NULL SET col = default`.

A escolha entre as duas estratégias é por probe (`CALL TABLE_INFO('Type')`)
em `_node_has_legacy_columns`.

A migração é **idempotente** — duas formas de cache:

* **In-memory** (`_MIGRATED_BOARDS: set[str]`) — populado após o primeiro
  pass bem-sucedido por board no processo.
* **Persistente** (`BoardMeta.schema_version`) — atualizado para `'0.3.0'`
  ao final da migração; o probe `_board_needs_v030_migration` retorna
  `False` em runs subsequentes mesmo após restart.

## Como executar

A migração roda **automaticamente** na primeira abertura de cada board
após o deploy: o gancho está em `BoardConnection.__init__`, que chama
`migrate_board_to_v030(board_id)` quando o probe detecta legacy columns
ou ausência das colunas v0.3.0 + BoardMeta sem `schema_version='0.3.0'`.

Para forçar manualmente (testes, CI, recovery):

```python
from okto_pulse.core.kg.schema import migrate_board_to_v030

summary = migrate_board_to_v030(board_id)
print(summary)
# {
#   'Decision': {'strategy': 'recreate', 'rows': 42},
#   'Criterion': {'strategy': 'alter', 'added': []},
#   ...
# }
```

## Defaults aplicados a nós legados

Boards migrados de v0.2.0 perdem o histórico binário de validação. Todo nó
recebe:

* `relevance_score = 0.5` (neutro — R2 recomputa logo após o primeiro
  trigger de scoring)
* `query_hits = 0` (R2 incrementa via hook no hybrid_search top-K)
* `last_queried_at = NULL`

A escolha de **0.5** (neutro) ao invés de 1.0 (alto) é deliberada — nós
legados entram no scoring sem privilégio histórico. O valor sobe ou desce
conforme uso real (R2 / R3).

## Stubs aplicados em R1

Para manter o servidor operacional entre R1 e R3, os filtros Cypher
`validation_status <> 'unvalidated'` foram **removidos** (não substituídos)
nas queries listadas abaixo. O comportamento temporário é "filtros
permissivos" — toda query retorna o grafo inteiro acima do
`min_confidence`, sem corte por relevância. R3 reintroduz o filtro
correto via `n.relevance_score >= $min_relevance`.

## Pendente para R2 (pipeline de scoring)

* `kg/scoring.py` (novo): `_compute_relevance(source_conf, degree, hits,
  contradict_conf, last_queried_at) -> float`.
* Hook em `commit_consolidation` para recálculo determinístico após
  novos nós/edges.
* Hit counter com lazy flush (10 hits OR 24h) em `kg_service`.
* Decay temporal half-life 30 dias aplicado **na leitura** (sem
  persistência intermediária).
* Evento SSE `kg.node.score_changed`.
* Métrica Prometheus histogram `okto_kg_relevance_score`.

## Pendente para R3 (integração + UI)

* Substituir os stubs Cypher pelos filtros reais
  `n.relevance_score >= $min_relevance`.
* `ORDER BY relevance_score DESC` em queries de ranking.
* Hook de hit counter no top-K do `hybrid_search`.
* Endpoint `POST /kg/boards/{bid}/nodes/{nid}/boost`.
* Frontend: `RelevanceBadge`, remoção do `ValidationStatus`,
  `KGValidationTab` recategorizada por faixa de score, slider
  `minRelevance` no `GraphControlsPanel`.
* Migração das fixtures dos 4 testes que ainda referenciam o nome
  antigo do campo (`tests/test_kg_*`).
