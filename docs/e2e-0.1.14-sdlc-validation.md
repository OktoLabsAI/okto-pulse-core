## Remaining Non-Conformities

### Automatic KG consolidation fails

The functional E2E passed, but automatic KG consolidation did not remain healthy.

Final health snapshot:

```text
queue_depth: 0
dead_letter_count: 13
total_nodes: 5
default_score_ratio: 0.0
```

Dead-letter rows failed with:

```text
KGPrimitiveError: commit failed and was compensated:
Binder exception: Trying to insert into an index on table Entity but its extension is not loaded.
```

`okto_pulse_kg_migrate_schema` returned success for the board, but the failed queue rows remained in DLQ.

Follow-up diagnosis: the schema/bootstrap path loaded the LadybugDB/Kuzu
VECTOR extension, but `BoardConnection` opened fresh hot-path connections
without loading it. Automatic consolidation then failed when inserting into
HNSW-indexed node tables, especially `Entity`. The follow-up fix loads the
VECTOR extension for every fresh `BoardConnection`.

### DLQ reprocess tool was not visible in this agent session

Tool discovery in the E2E agent session exposed `okto_pulse_kg_dead_letter_list` as read-only and stated that MCP reprocess was deferred. Source-level inspection later confirmed that `okto_pulse_kg_dead_letter_reprocess` is registered in the MCP server; this should be treated as a tool discovery / installed metadata freshness issue if it recurs after rebuild and restart.

### Consolidated traceability report tool was not visible in this agent session

No dedicated MCP report was visible to the E2E agent session for:

```text
ideation -> refinement -> spec -> sprint -> card -> test/bug -> artifacts
```

Traceability had to be reconstructed from `get_ideation_context`, `get_refinement_context`, `get_spec_context`, sprint context, card context, validations, task conclusions, and KG queries. Source-level inspection later confirmed that `okto_pulse_get_traceability_report` is registered in the MCP server; if it is still not discoverable after rebuild/restart, inspect the MCP tool metadata cache.

### Some MCP tool descriptions are stale

Observed description mismatches:

- `okto_pulse_create_card` still says the spec must be `done`, although test cards were accepted after validation and before final spec completion.
- `okto_pulse_create_card` still documents `card_type` as `normal` or `bug`, while `test` is accepted and required for test/regression flows.
- `okto_pulse_submit_task_validation` still says failed validation routes to `not_started`; the desired and observed behavior is to remain in `validation`.
- `okto_pulse_kg_dead_letter_list` still says DLQ is read-only and reprocess is deferred.

## Follow-Up Recommendations

1. Rebuild/reinstall and re-run KG automatic consolidation to verify the `BoardConnection` VECTOR extension fix against real DLQ rows.
2. Verify that `okto_pulse_kg_dead_letter_reprocess` is discoverable after rebuild/restart and use it after schema/index fixes.
3. Verify that `okto_pulse_get_traceability_report` is discoverable after rebuild/restart and use it for the full SDLC chain.
4. Update MCP descriptions for `create_card`, `submit_task_validation`, and KG DLQ tools to match actual behavior.
5. Keep deterministic KG edge ownership reserved for the worker and document that cognitive agents must use only allowed cognitive edge types.

## Final Assessment

The application SDLC workflow, gates, artifact propagation, task validation, bug/regression handling, and manual KG queryability are working well enough for the validated product flow.

The main blocker is the automatic KG consolidation path. It currently fails for several artifacts with an index-extension error and leaves DLQ entries that cannot be reprocessed through a visible MCP tool.
